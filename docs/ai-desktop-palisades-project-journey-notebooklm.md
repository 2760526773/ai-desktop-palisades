## 一、项目起点

项目最初的目标不是做一个普通的桌面美化工具，而是做一个“能真正整理 Windows 桌面”的轻量程序。

用户最初提出的核心需求包括：
- 识别 Windows 桌面程序和文件类型
- 用 AI 模型自动判断分类，如娱乐、影音、游戏、文档、工具等
- 将同类项目归档到类似“桌面栅栏”的区域中显示
- 界面风格希望是毛玻璃、透明、接近谷歌风格
- 程序要轻量，占用小，可开机自启动
- 最好像已有桌面栅栏工具那样直观，但减少手工操作

一开始的理解方向偏向“做一个新的桌面整理程序原型”，因此先尝试了 Python + Tk、再尝试 Python + PySide6 的路线，希望更快做出可运行原型。

但在推进过程中，需求逐渐收敛为：
- 不只是做一个新工具
- 而是要尽量接近现有成熟桌面栅栏产品的交互形态
- 同时加入 AI 自动分类这个核心差异点

这让项目从“快速原型”转向了“基于现有桌面栅栏项目做实用增强版”的路线。

## 二、第一阶段：Python 原型路线与放弃原因

### 1. 早期尝试

最早尝试的是一个 Python 原型，目的是快速验证以下事情：
- 能否扫描桌面第一层对象
- 能否把对象归类后显示在一个栅栏界面里
- 能否在界面中切换 AI 提供商并填写 API Key
- 能否做到基本的桌面分栏视觉

先后经历了：
- Tk 原型
- PySide6 原型

### 2. 这条路线遇到的问题

这条路线可以证明思路可行，但很快暴露出几个现实问题：

- 与 Windows 桌面原生交互不够贴近
- 快捷方式、系统右键、桌面拖拽这些行为实现成本高
- 多语言路径、控制台编码、子进程读取时容易出现编码问题
- UI 容易做出“看起来像 demo”的效果，但离“长期使用的桌面工具”差距很大
- 和已有开源桌面栅栏项目相比，重复造轮子太多

在这个阶段还出现了一个具体报错：
- Python 子线程读取控制台输出时触发 `UnicodeDecodeError`
- 这暴露出 Windows 本地编码、控制台编码与 Python 默认 UTF-8 解码之间的兼容性问题

### 3. 路线调整

当用户明确指出“方向错了”，并给出 GitHub 项目 `Xstoudi/Palisades` 作为参考后，路线正式切换。

新的目标变成：
- 基于 Palisades 这类桌面栅栏项目继续开发
- 保留类似桌面栅栏的体验
- 新增 AI 自动分类能力
- 尽量做成轻量、可发布、可日常使用的 Windows 程序

## 三、第二阶段：切换到 Palisades 作为上游基础

### 1. 上游选择

选定的上游项目是：
- `Xstoudi/Palisades`
- GitHub 地址：https://github.com/Xstoudi/Palisades

选择它的原因很直接：
- 已经具备桌面栅栏的基本交互模型
- 是 WPF 桌面程序，更贴近 Windows 桌面原生体验
- 比从零写一个桌面栅栏更现实
- 可以把开发重点放在 AI 分类和整理流程上，而不是重新搭一整套桌面管理框架

### 2. 最初遇到的构建问题

克隆后立即遇到了构建层面的问题。

首先，使用 `dotnet build` 时出现了 COM 引用相关错误：
- `ResolveComReference` 不受当前 .NET Core 版本的 MSBuild 支持

随后在继续调整过程中，还出现了典型的 C# 编译错误，例如：
- `string[]` 上的 `Contains` 扩展方法解析问题
- `AiSettings` 命名冲突，位于不同命名空间中
- 若干 nullable warning

这些问题说明：
- 上游项目本身并不是“拉下来直接可无脑改”的状态
- 需要先让基础工程进入可稳定构建状态，后续才能继续做功能修改

### 3. 先把项目拉回“能编译、能运行”

这一阶段做的事情主要是：
- 修复命名冲突
- 调整缺失的 `using`
- 修复编译器不再自动推断的旧用法
- 保持 nullable warning 暂时存在，但先保证主工程可运行

当 `dotnet build` 和 `dotnet run` 首次成功后，项目才真正进入功能开发阶段。

## 四、核心产品方向确定

随着讨论深入，项目的产品定义逐渐明确下来。

最终确定的方向不是“桌面图标展示器”，而是：

**一个基于桌面栅栏交互的 AI 桌面整理工具。**

核心定义包括：
- 处理对象为桌面第一层的文件、文件夹、快捷方式
- 用 AI 模型识别项目类型
- 按分类放进对应栅栏
- 支持从桌面拖入栅栏、从栅栏拖回桌面
- 支持从一个栅栏拖到另一个栅栏改分类
- 支持真实物理归档，而不是只做镜像显示
- 支持退出软件时恢复桌面对象

### 明确排除的内容

也有一些范围被明确收紧：
- 不追求 1:1 复刻商业桌面栅栏软件的全部细节
- 不恢复系统桌面图标像素级坐标
- 只处理桌面第一层，不做递归深层整理
- 优先保证“能整理、能恢复、能使用”，美化排在后面

这个定义很重要，因为它决定了后面的所有实现取舍。

## 五、AI 分类能力的引入

### 1. 初始目标

用户希望：
- 软件能接入主流 AI 模型
- 不需要手改配置文件
- 可以在界面中切换提供商
- 输入 API Key 即可使用
- 支持诸如 GPT、Kimi、Doubao、Gemini、DeepSeek 等模型

### 2. 支持的提供商逐步扩展

最终项目加入了多提供商 AI 支持，包含：
- OpenAI
- Kimi / Moonshot
- Doubao / Ark
- DeepSeek
- Gemini
- Qwen
- Groq
- Grok
- OpenRouter
- Mistral
- Custom

并加入了：
- 提供商切换下拉框
- API Key 输入
- Base URL 配置
- 推荐模型下拉框
- 手动输入模型名的能力

### 3. AI 分类流程的演进

最初的 AI 分类流程比较粗糙，后来逐步演进为：
- 扫描桌面项目
- 抽取名称、扩展名、快捷方式目标等基本信息
- 构造给模型的精简提示词
- 请求模型返回分类结果
- 将返回结果归一化为固定类别
- AI 失败时回退到本地启发式规则

### 4. 关键问题与修复

AI 接入过程中出现了几类典型问题：

#### 问题一：界面看不到 AI 是否真的在工作
修复方式：
- 增加明确的成功/失败弹窗
- 增加“立即AI分类”入口
- 保存 AI 设置后立即可验证

#### 问题二：模型返回 400，token 超限
示例问题：
- `requested 13946, model limit 8192`

修复方式：
- 改为自动分批分类
- 给模型的输入改为精简结构，而不是完整路径大 JSON
- 支持按 `id -> category` 的形式返回

#### 问题三：后台线程直接改 WPF 绑定集合，触发线程访问异常
示例错误：
- “调用线程无法访问此对象，因为另一个线程拥有该对象。”

修复方式：
- 后台线程只做扫描和分类计算
- 回到 UI 线程后再统一应用分类结果

#### 问题四：重复分类导致结果堆叠
修复方式：
- 每次 AI 分类前，先清理旧的 AI 分类栅栏内容
- 重建分类栏，避免历史结果残留
- 对栅栏内项目再次去重

#### 问题五：桌面不变，只是复制显示
修复方式：
- 从“镜像展示”改为“物理归档”
- 桌面对象真实移动到受管目录后，再在栅栏里显示

## 六、桌面物理归档与恢复机制

这是项目和普通“桌面分组显示器”最本质的差异之一。

### 1. 物理归档目标

用户明确要求：
- 真正把项目分类到栅栏里
- 桌面实际内容要变化
- 不是在栅栏里复制一份图标而已

因此项目加入了受管目录机制：
- 将桌面对象按分类移动到应用管理目录中
- 栅栏显示的是这些真实对象
- 支持从桌面拖进栅栏，也支持移回桌面

### 2. 退出恢复

后续又追加了一个关键需求：
- 退出软件后，桌面要恢复

于是补上了恢复逻辑：
- 应用退出时扫描受管目录
- 将归档对象移动回桌面根目录
- 如果桌面已有同名对象，则自动重命名避免覆盖

### 3. 这个机制的意义

这套机制解决了两个现实问题：
- 软件不会永久“绑架”用户桌面
- 用户可以把它当作“动态整理工具”，而不是“必须常驻才能使用的壳层”

## 七、拖拽、右键、桌面式交互

用户非常强调一点：

**栅栏里的项目必须像正常桌面图标那样能用。**

这导致项目必须超越“一个分类面板”，而走向“桌面式交互容器”。

### 1. 最终加入的交互能力

逐步实现了：
- 左键打开项目
- 右键常用菜单
- 打开所在位置
- 删除
- 查看属性
- 从桌面拖入栅栏
- 从栅栏拖回桌面
- 栅栏之间拖拽改分类
- 鼠标悬停显示名称和路径

### 2. 右键菜单的反复调整

这一块踩了很多坑：
- 自定义样式导致菜单项被裁切
- 自定义模板导致右键唤不出来
- WPF 菜单重样式后容易破坏系统默认行为

后来的结论很务实：
- 对桌面式操作，稳定性比“花哨样式”更重要
- 最终菜单样式一度回退到更保守的系统默认渲染方式
- 先保证功能完整，再谈视觉统一

这是整个项目中的一个典型取舍：
- 不追求一次把 UI 做到最漂亮
- 先把桌面行为做正确

## 八、UI 风格的多轮试错

UI 是整个开发过程中来回调整最多的部分之一。

### 1. 用户最初的期待

用户希望：
- 毛玻璃
- 透明
- 谷歌风格
- 圆角润滑
- 视觉轻量

同时给出了原型图，强调：
- 想要右上角那种透明桌面栅栏效果
- 蓝色来自壁纸，栅栏本体应该透明
- 不希望是大片灰块或黑块

### 2. 试错过程

先后尝试过：
- 黑色玻璃风
- 大卡片圆角风
- 毛玻璃 + 分层透明窗口
- 扁平浅色菜单
- 半透明色块

但这些方案都暴露出不同问题：
- 不够像用户给的原型
- WPF 分层窗口带来性能问题
- 菜单和栅栏风格不统一
- 自定义模板容易把功能做坏
- 用户更关心“像原型”和“实用”，而不是“设计师风格”

### 3. 最终方向

最后栅栏主体回到了更接近原型的透明风格：
- 透明栅栏底板
- 细边框
- 桌面上轻量悬浮
- 不再强推厚重玻璃或花哨装饰

同时加入了：
- 左上角折叠/展开
- 右上角锁定
- 菜单入口

### 4. 一个重要教训

在这个项目里，UI 的经验非常明确：
- 自定义外观不能破坏桌面交互
- 菜单稳定性优先于花样模板
- 透明效果不能以明显卡顿为代价
- 视觉还原要服从用户给的原型，而不是开发者主观审美

## 九、性能与启动流程优化

随着功能变多，启动卡顿成为一个明显问题。

### 1. 卡顿原因

用户发现：
- 每次启动软件都会卡住一段时间

分析后确认原因主要来自：
- 启动时自动执行 AI 分类
- 同时扫描桌面
- 提取图标
- 更新布局
- 触发 UI 渲染

### 2. 后续优化决策

于是做了三项调整：
- 默认不自动 AI 分类
- 启动逻辑改后台异步，尽量不阻塞界面
- 增加桌面状态缓存，桌面未变化时跳过重复分类

这几项优化的目标很明确：
- 打开软件先能用
- AI 分类变成主动触发，而不是每次启动都强制跑
- 对日常场景，减少重复成本

## 十、重复图标与脏数据清理

这是项目中非常典型的一类“真实使用才暴露出来”的问题。

### 1. 重复图标出现的原因

用户在多开或多次整理后发现：
- 同一个项目会在多个栅栏重复出现
- 有些旧分类栏内容残留
- 桌面和栅栏之间状态不一致

### 2. 修复思路

后来通过多种方式逐步降低这个问题：
- 加单实例锁，防止多个程序实例并发操作桌面状态
- 清理旧的 AI 分类结果再重建
- 对栅栏内项目做去重
- 对历史保存布局做脏数据清洗
- `.lnk` 解析失败时降级处理，避免因为解析异常产生重复对象

### 3. 结果

虽然重复问题属于桌面类工具里比较麻烦的一类边界问题，但项目最终已经具备了：
- 明显更稳定的重复控制
- 更干净的分类重建流程
- 避免历史状态越积越乱的清理逻辑

## 十三、重新打包发布时遇到的问题

在生成 `publish` 目录和最终发布包时，也出现了几个 Windows 桌面项目常见问题。

### 1. `publish` 目录删除失败

原因：
- 正在运行的 `Palisades.exe` 占用了发布目录内文件

处理方式：
- 查找运行中的 `Palisades` 进程
- 停止相关进程
- 删除旧 `publish`
- 重新执行 `dotnet publish`

### 2. `dotnet publish` 在沙箱环境下失败

原因包括：
- 首次运行时想写入默认用户目录
- 无权访问用户目录下的 NuGet 配置文件

处理方式：
- 设置本地 `DOTNET_CLI_HOME`
- 最终在提权环境下执行 publish
- 让发布过程能够正确读取 NuGet 配置并完成构建

### 3. 最终结果

最终成功生成：
- `publish/Palisades.exe`
- `publish/AI-Desktop-Palisades-v0.3.1-win-x64.zip`

说明项目已经具备完整的本地打包和发布条件。

## 十五、项目最终形态（当前阶段）

截至当前版本，项目已经从最初的模糊需求，逐步形成了一个明确可运行的产品形态。

### 当前可以概括为：

**AI Desktop Palisades 是一个基于桌面栅栏交互、支持主流 AI 模型自动分类、可真实物理整理桌面对象，并在退出时恢复桌面的 Windows 桌面整理工具。**

### 当前主要能力包括：
- 透明桌面栅栏界面
- 桌面第一层对象扫描
- AI 自动分类
- 多提供商切换
- 推荐模型下拉
- 物理归档
- 从桌面拖入栅栏
- 从栅栏拖回桌面
- 栅栏间拖动改分类
- 左键/右键/悬停等桌面式交互
- 退出恢复桌面对象
- GitHub 可发布版本

### 当前已知边界包括：
- 并非完全复刻商业桌面栅栏软件
- 右键菜单仍以稳定可用为优先，不追求完全原生 Explorer 全量实现
- 某些 AI 提供商需要用户自己拥有可用模型 ID 或 endpoint
- 透明桌面类 UI 在 WPF 下仍然需要谨慎权衡样式与性能

## 十六、关键决策摘要

为了方便 NotebookLM 提问时快速定位，这里将最重要的决策浓缩列出。

### 产品决策
- 不做纯展示型栅栏，要做真实桌面整理工具
- 不做 AI 炫技，要做减少手动分类的实用功能
- 不追求完全复刻 Fences，而是做 AI 增强版桌面栅栏

### 技术决策
- 放弃 Python 原型作为最终方案
- 选择 WPF 上游项目 Palisades 作为基础
- AI 适配多提供商，配置界面化
- AI 分类采用“用户操作触发 + 缓存 + 回退启发式”的策略
- 桌面对象采用物理归档，而不是镜像复制
- 退出时恢复桌面，避免软件长期劫持桌面状态

### 交互决策
- 栅栏里的项目必须像桌面图标一样能操作
- 拖拽和右键是核心，不是附加功能
- 菜单样式可保守，但功能必须完整
- UI 风格服从用户提供的参考原型

### 发布决策
- 重写 README，明确上游来源和改动内容
- 生成压缩包供 GitHub Release 使用
- 使用语义版本进行标签管理

## 十八、结论

这个项目不是一次性按计划顺推完成的，而是在不断试错中逐步收敛出来的。

它经历了：
- 技术路线误判
- Python 原型试错
- 切换上游项目
- 编译修复
- AI 接入与 token 问题
- 多线程 UI 崩溃
- 重复图标和脏数据问题
- 物理归档与恢复机制完善
- UI 反复推翻重做
- Git、代理、发布流程打通
- README 和 Release 准备
- 外部传播文案整理

最终得到的不是一个“看起来很炫”的演示项目，而是一个已经具备实际使用价值的 Windows 桌面整理工具。

它的核心价值可以总结为一句话：

**把传统桌面栅栏的交互方式，和 AI 自动分类能力结合起来，让桌面整理从手工劳动变成半自动流程。**
